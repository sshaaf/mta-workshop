== 1. Goals of this lab

As enterprise organizations accelerate their cloud adoption journey, one of the most significant challenges they face is modernizing legacy Java applications to run on modern platforms like Red Hat OpenShift. The traditional approach to code refactoring is time-consuming, error-prone, and requires deep knowledge of both legacy and target technologies.

The goal of this lab is to experience how **Red Hat Developer Lightspeed for Migration Toolkit for Applications (MTA)** revolutionizes application modernization by combining static code analysis with generative AI to accelerate the migration of JavaEE 7 applications to **Quarkus**.

This hands-on lab covers the following topics:

* Configure the MTA extension in Red Hat Dev Spaces, including Developer Lightspeed (AI-assisted migration)
* Analyze the application and receive AI-powered code refactoring recommendations
* Apply intelligent fixes and migrate the application to Quarkus and Jakarta EE.
* Deploy and validate the modernized application on OpenShift

== 2. Introduction to Developer Lightspeed for MTA

Migration Toolkit for Applications has long been the industry-leading solution for assessing and analyzing applications for cloud migration. **Developer Lightspeed** is MTA's **AI-powered code refactoring** feature—not a separate extension. It brings generative AI-assisted migration into the developer's IDE, transforming how organizations approach application modernization.

=== 2.1. The Challenge of Application Modernization

Enterprises face several critical challenges when modernizing their application portfolios:

* **Security concerns** - Outdated applications and frameworks introduce significant risks to business operations, potentially leading to data breaches and compliance violations.
* **Complexity of legacy systems** - Legacy codebases often lack compatibility with modern platforms, have tightly coupled dependencies, and may be unfamiliar to current development teams.
* **Competing priorities** - Organizations must balance the urgency of modernization with limited resources, often prioritizing new features over improving existing applications.
* **Scale and consistency** - Large enterprises may have hundreds of applications requiring similar changes, making manual migration impractical.

=== 2.2. How Developer Lightspeed Transforms Modernization

**Red Hat Developer Lightspeed for MTA** denotes MTA modernization that uses both static analysis and **Developer Lightspeed** (generative AI). In the IDE, the **MTA extension** for Visual Studio Code provides MTA static analysis and Developer Lightspeed together. That combination addresses these challenges:

* **Speed up application modernization with GenAI** - The MTA extension applies MTA's proven static code analysis together with Developer Lightspeed generative AI to identify migration issues and propose context-aware solutions. The AI generates tailored code recommendations directly within the IDE, eliminating context switching and keeping developers focused on their work. The system handles ripple effects of code changes automatically, maintaining code integrity throughout the refactoring process. Additionally, it tracks migration metrics and learns from previous solutions to continuously improve recommendations over time.


* **Get helpful refactored code suggestions developers trust** - The system uses rule-based analysis to guide the LLM on what migration issues to look for and how to solve them, ensuring recommendations are grounded in proven migration patterns. Every suggestion includes clear explanations of what needs to change and why, helping developers understand the reasoning behind each proposed fix. Real-time IDE diagnostic data ensures fixes address actual compilation and runtime issues rather than hypothetical problems. Throughout the process, developers review and approve changes, maintaining full control over the refactoring process and ensuring code quality.


* **Modernize custom and proprietary technologies** - Developer Lightspeed enables creating custom rules that specify conditions and actions for organization-specific frameworks, allowing the AI to understand and work with a unique technology stack. This capability automates code transformation for applications using proprietary or corporate libraries that aren't covered by standard migration rules. Extend the out-of-the-box migration paths with organizational knowledge, making the tool adaptable to specific business needs.


* **Scale modernization across the portfolio** - Developer Lightspeed learns from changes made during the migration process, building institutional knowledge that improves over time. As it encounters similar patterns across applications, it accurately predicts and generates solutions for similar applications across the portfolio. This learning capability enables refactoring multiple applications predictably and efficiently with consistent quality, making portfolio-scale modernization achievable.


* **Seamless integration into development workflows** - All analysis and refactoring happens directly in Red Hat Devspaces, keeping developers in their familiar environment. Detailed issues, proposed solutions, and code diffs are visible without switching contexts or jumping between different tools. Once refactoring is complete, push the refactored code through existing CI/CD pipelines for automated testing and deployment, ensuring the modernized application integrates seamlessly with established development processes.


* **Choose an AI model** - Developer Lightspeed supports using a custom LLM—public or self-hosted—with no fine-tuning required, eliminating vendor lock-in. The RAG architecture learns from past solutions and stores them in a knowledge base, continuously improving its recommendations based on the organization's migration history. The system is compatible with multiple LLM providers through OpenAI-compatible APIs, including Red Hat OpenShift AI (enterprise model builder & inference server), Red Hat Enterprise Linux AI (enterprise inference server), and vLLM (popular enterprise inference server). This flexibility supports balancing performance, cost, and data privacy according to specific requirements.

== Exercise 3: Modernize Message-Driven Bean (MDB) to Reactive Messaging

Modernizing legacy "Enterprise Java" (Java EE/Jakarta EE) applications is often a manual, error-prone process. This exercise guide focuses on **Developer Lightspeed**—the AI feature of MTA, delivered through the MTA extension—which combines static code analysis with Large Language Models (LLMs) to automate the refactoring of legacy patterns into modern, cloud-native **Quarkus** code.

=== exercise 1: Modernize Message-Driven Bean (MDB) to Reactive Messaging
In this exercise, let's transform a legacy `OrderServiceMDB` into a modern Quarkus service using SmallRye Reactive Messaging.

==== The Architectural Shift: Synchronous vs. Asynchronous
Legacy Java EE relies on **JMS (Java Message Service)**. While technically asynchronous, JMS in an MDB context often operates in a "blocking" mindset. The MDB container requires heavy resource-consuming threads to sit and wait for messages to arrive.

**Reactive Messaging** (via brokers like Kafka) is built on the **Reactive Streams** standard. Instead of a thread "waiting," the system uses a non-blocking stream. When a message arrives at a Kafka topic, it is "pushed" through a functional pipeline.

==== Why Cloud-Native?
* **Resource Efficiency:** Thousands of messages are handled with only a few CPU threads.
* **Elastic Scaling:** Reactive services scale up and down rapidly in Kubernetes because they lack the "baggage" of an EJB container.
* **Developer Experience:** Quarkus replaces dozens of lines of XML or complex `@ActivationConfigProperty` with a simple `@Incoming` annotation.

==== Step-by-Step with Developer Lightspeed
*1*. The first step is to review the Analysis from our previous exercise.
In the `Analysis panel` search box type the class name `OrderServiceMDB`.

image::mta-dls/orderservicemdb-analysis-results.jpg[OrderServiceMDB,link=self,window=blank,title="OrderServiceMDB"]

Let's review the issues the static code analyzer has reported. The issues are reported in a list as shown in the image above. Click on the `>` icon to get more details. You should be able to see the following issues reported.

- `@MessageDrivenBean` - EJBs are not supported in Quarkus.
  Enterprise Java Beans (EJBs) are not supported in Quarkus. CDI must be used. Please replace the @MessageDriven annotation with a CDI scope annotation like @ApplicationScoped.
- Configure message listener method with `@Incoming`. 
  The destinationLookup property can be migrated by annotating a message handler method (potentially onMessage) with the org.eclipse.microprofile.reactive.messaging.Incoming annotation, indicating the name of the queue as a value. For example:
[.console-input]
[source,java,subs="+attributes,macros+"]

  @ApplicationScoped
  public class OrderService {
      @Incoming("orders") 
      public void process(Order order) {
          // Business logic here
      }
  }
  ----
- JMS is not supported in Quarkus.
  References to JavaEE/JakartaEE JMS elements should be removed and replaced with their Quarkus SmallRye/Microprofile equivalents.
- The package `javax` has been replaced by `jakarta`.
  Replace the javax.jms import statement with jakarta.jms

== Exercise 4: Transforming Legacy EJBs to REST Endpoints

This exercise covers the transformation of a `ShippingService` from a Stateless Session Bean (EJB) into a modern RESTful API.

=== The Paradigm Shift: RMI to HTTP
Legacy **EJBs** were designed for **RMI (Remote Method Invocation)**. This required brittle "Remote Interfaces" and tight coupling between the client and the server.

**REST (Representational State Transfer)** uses standard HTTP. It is the universal language of the cloud, allowing any service (Python, Go, JS) to communicate with the Java backend without special client libraries.

==== Benefits of Going Cloud-Native
* **Decoupling:** Services are found via URLs, not complex JNDI lookups.
* **Observability:** Modern tools like Service Meshes (Istio) can monitor HTTP traffic easily; RMI is largely "invisible" to cloud infrastructure.
* **Simplicity:** Refactoring moves from the complex EJB lifecycle to a simple `@ApplicationScoped` CDI bean.

==== Step-by-Step with Developer Lightspeed
1. **Identity:** flags the `@Stateless` and `@Remote` annotations.
2. **Transform:** Developer Lightspeed moves business logic into a POJO (Plain Old Java Object).
3. **Expose:** Developer Lightspeed generates JAX-RS annotations like `@Path("/shipping")` and `@GET`.
4. **Result:**
[source,java]

// Modernized REST Endpoint
@Path("/shipping")
@ApplicationScoped
public class ShippingService {
    @GET
    @Produces(MediaType.APPLICATION_JSON)
    public double calculateCost(@QueryParam("weight") int weight) {
        return weight * 1.5;
    }
}


== Exercise 5: Enforcing Standards with the Solution Server

This exercise covers how to "teach" the AI the organization’s specific internal standards using the **Solution Server** (used with Developer Lightspeed in the MTA extension).

=== The Problem: Generic AI vs. Enterprise Reality
General AI might suggest a standard logger like `Log4j`. However, large enterprises usually have a specific **Internal Audit Library** required for compliance. 

==== The Solution: The "Learning" Migration
The **Solution Server** acts as "Shared Memory" for the modernization team. It captures human decisions and applies them to future AI generations.

==== The Architectural Benefit
* **Consistency:** Every application in the portfolio is modernized in the exact same way.
* **Compliance:** Security and logging standards are baked into the migration pipeline.
* **Acceleration:** As the server "learns," it reduces the need for developers to manually correct the AI.

==== Step-by-Step with Developer Lightspeed
1. **First Fix:** Developer Lightspeed suggests a generic logger. Manually edit it to use the organization's `InternalAudit.log()`.
2. **Record Hint:** **Accept** the fix. Developer Lightspeed records this pattern in the **Solution Server** database.
3. **Automate:** When opening the next legacy file, Developer Lightspeed checks the Solution Server first.
4. **Result:** Developer Lightspeed now generates the company-approved `InternalAudit` code on the first try, ensuring all 100+ applications in the portfolio follow the "Golden Path."
