In this part of the workshop, ACME wants to **scale** modernization while keeping **company-specific** patterns and policies consistent. **Solution Server** records accepted fixes and surfaces them as **Migration Hints** so junior developers and contractors get the same guidance as your lead architects.

IMPORTANT: Solution Server is an enterprise-oriented capability: it applies **Migration Hints** to the LLM when generating fixes, using policies and **tribal knowledge** captured as developers accept and adjust solutions.

In later exercises you will **switch roles**—from architect-style adjustments to **developer** flows that **consume** hints. This exercise only **introduces** Solution Server and **prepares** the MTA extension, Hub, and profile.

== Introduction to Developer Lightspeed Solution Server

You need to understand what Solution Server is and how it enables organizational knowledge capture for AI-assisted development, before you connect the IDE to the Hub and run analyses that use it.

=== Prerequisites

* High-level understanding of the value of **Developer Lightspeed for MTA** (earlier exercises in this lab)
* Access to the workshop **OpenShift Dev Spaces** workspace and application tabs

=== Steps

. Understand Solution Server's role:

Developer Lightspeed Solution Server acts as an organizational knowledge repository that:

* Captures company-specific coding patterns and standards
* Learns from existing codebase examples
* Stores best practices and policy rules
* Provides context to AI models for consistent code generation
* Enables knowledge sharing across development teams

. Review the Solution Server architecture:

Solution Server is an **MCP server** on top of MTA's deployment. It listens to **before** and **after** code changes during Developer Lightspeed–driven migration, reconstructs them as **hints**, and makes them available to developers connected to the **MTA Hub** and Solution Server. When you run **Analysis**, rule-enriched prompts go to the LLM, which can fold in **Migration Hints** from past accepted fixes.

This flow helps keep AI-generated changes aligned with **organizational** standards.

. Explore key Solution Server capabilities:

* **Policy enforcement** — Define expectations the AI should follow
* **Pattern recognition** — Learn from existing code examples
* **Context injection** — Organize knowledge in AI requests
* **Consistency** — Similar assistance for all team members
* **Knowledge evolution** — Improve as more patterns are stored

. Consider ACME’s use cases:

* Standardized error handling
* Logging conventions
* Security and configuration policy
* Testing and documentation standards

=== Verify

Confirm you can explain:

* Solution Server **captures** organizational knowledge
* It **adds context** to AI models for code generation
* **Policies and patterns** apply **consistently**
* Teams **share** knowledge through centralized hints

== Ensure access and configuration

Set up the workspace and MTA extension so the **Hub**, **GenAI**, and **analysis profile** are ready for Solution Server use.

=== Prerequisites

* The concepts from *Introduction to Developer Lightspeed Solution Server* (above)
* Access to the workshop **Dev Spaces** workspace

=== Steps

. Confirm access to the Dev Spaces workspace

There are two browser contexts: the **Dev Spaces** list (all workspaces) and the **opened** workspace (VS Code in the browser). If you still have a tab from xref:module-mta-developer-lightspeed-exercise-1.adoc[Exercise 1], switch to it. If not, use the **OpenShift Dev Spaces** (or equivalent) entry in the workshop interface and open your workspace, signing in with your lab credentials as documented in Exercise 1.

. Switch to the branch for the Solution Server track

* Open a new terminal: hamburger menu → **Terminal** → **New Terminal**
* If your environment provides a dedicated branch (for example `module3` in the *Red Hat* Solution Server module), check it out:

[source,bash,role="execute"]
----
git check$out module3
----

image::mta-dls/exercise-5/git-checkout-module3.jpg[Checkout module branch,link=self,window=blank,title="Checkout module branch"]



You can close the terminal panel if you no longer need it.

. Confirm the MTA (Developer Lightspeed) extension

If the **MTA Analysis View** is already open from xref:module-mta-developer-lightspeed-exercise-1.adoc[Exercise 1], use it. Otherwise, open the **MTA** activity and the **MTA Analysis View** as in Exercise 1 (install/load the **MTA** extension if your environment requires it).

. Connect to the **MTA Hub** and confirm access to the **LLM**

The Hub connection enables **Solution Server** (as an MCP server) together with analysis.

* In the **MTA Analysis View**, you may see connection warnings temporarily; you can still configure the Hub. Click the **configuration (gear) icon** on the upper right.

image::mta-dls/exercise-5/Analysis-panel-initial.jpg[Open configuration,link=self,window=blank,title="Open configuration"]

* The **Get Ready to Analyze** side panel opens.

image::mta-dls/exercise-5/Analysis-Config.jpg[Analysis configuration,link=self,window=blank,title="Analysis configuration"]

* **Hub** may be unset and **GenAI** may need configuration.
* Click **Configure Hub Settings** to open **MTA Hub Configuration**.
* In **General Settings**, enable the slider for **connection to Konveyor Hub (MTA Hub)**.
* In **Hub URL** enter:

[source,bash,subs=attributes]
----
{mta_url}
----

* Enable **Skip SSL certificate verification** if your environment requires it.
* Under **Features**, enable **Solution Server**.
* Click **Save** and close the **MTA Hub Configuration** tab when done.

image::mta-dls/exercise-5/Hub-Config1.jpg[Hub configuration,link=self,window=blank,title="Hub configuration"]
image::mta-dls/exercise-5/Hub-Config2.jpg[Hub configuration continued,link=self,window=blank,title="Hub configuration continued"]

If **Configure GenAI** is *not* yet configured:

* Click **Configure GenAI Settings** so `provider-settings.yaml` opens.

image::mta-dls/exercise-5/provider-settings.jpg[AI model settings,link=self,window=blank,title="AI model settings"]

* Replace the block from `models:` through the **first** `OpenAI` (or first) model entry with the following (keep YAML indentation):

[source,yaml,role="execute",subs=attributes]
----
models:
  OpenAI: &active
    environment:
      OPENAI_API_KEY: "{litellm_virtual_key}" # Required
    provider: ChatOpenAI
    args:
      model: llama-scout-17b # Required
      configuration:
        baseURL: "{litellm_api_base_url}"
----

* Save/close `provider-settings.yaml` as appropriate for your editor, close **Get Ready to Analyze** and any extra Hub panels, and return to **MTA Analysis View**.
* Ensure **Server status** is **Running**; use **Start** if needed.
* For this track, keep **Agent Mode** *off* unless an exercise explicitly enables it.

image::mta-dls/exercise-5/Server-Status.jpg[Server status running,link=self,window=blank,title="Server status running"]

. Review the **active profile**

A profile is often pre-provisioned for the Solution Server scenario. Here you only **verify** the profile you will use for analysis; multiple profiles are normal in real migrations (different source/target stacks and rule sets).

* Open **MTA Analysis View** again.

[IMPORTANT]
====
For this walkthrough, keep **Agent Mode** *off* so the lab steps stay within time. Keep **Server status** **Running**.

image::mta-dls/exercise-5/Server-Status.jpg[Server status,link=self,window=blank,title="Server status"]
====

* Open the **profile** dropdown and choose **Manage Profiles** if you need to inspect or activate a profile.

image::mta-dls/exercise-5/Open-Profiles.jpg[Open profiles,link=self,window=blank,title="Open profiles"]

* Find the profile intended for this module (for example **Solution Server** in the *Red Hat* lab). Open it and confirm it is the **active** profile.

image::mta-dls/exercise-5/Active-Profile.jpg[Active profile,link=self,window=blank,title="Active profile"]

** Target and source technologies should match your **coolstore** / lab scenario. 
** The profile typically includes **default** MTA rules plus any **custom** rules your org added (for example company-specific logging checks). 
* Close the **Manage Profiles** panel.

[NOTE]
====
Authoring custom rules in depth is out of scope for this workshop; ask your team or instructor for enablement.
====

=== Verify

* You can reach **MTA Analysis View** with the extension running.
* **MTA Hub** is configured (including **Solution Server**), with **LLM** settings in `provider-settings.yaml` as above (or an equivalent your instructor approves).
* The **analysis server** is **Running**; **Agent Mode** is *off* unless an exercise says otherwise.
* A **profile** (for example *Solution Server*) is **active** for the next analyses.
