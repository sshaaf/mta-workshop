= Exercise 2: Run an initial static code analysis
:imagesdir: ../assets/images/

[.goal]
.Goal
[NOTE]
====
Run an initial **MTA static code analysis** over the coolstore repository so you get a structured list of migration **issues**, links into source files, and rule-based context— the baseline you will use in later exercises to drive **Developer Lightspeed** and generated fixes.
====

Before making changes, use the MTA extension to analyze the legacy code and identify specific modernization opportunities.

In this exercise the MTA extension runs a static code analysis against all the source code in the cloned repository. The static code rules are provided by Red Hat to help focus on rapid identification of issues, and customer rules can also be written and included.In this case we are including the custom ACME specific company policy code rules. This provides an efficient starting point focused on identifying migration issues that were highlighted during the risk analysis in the last module.

Later on the analysis results will be used to prompt the LLM for generating code fixes.

=== Prerequisites
* Dev Spaces workspace running with the MTA extension configured (including Developer Lightspeed)
* Repository **Developer Lightspeed-coolstore** successfully cloned

=== Steps

. Start the Analysis Server
+
* Navigate to the **MTA Analysis View** pane
+
TIP: If the MTA Analysis View panel is closed accidentally, click the MTA Extension icon and click the book icon in the pane to the right at the top. image:mta-dls/exercise-2/reopen-analysis-pane.jpg[Reopen View Panel,link=self,window=blank,title="Reopen View Panel",150,150]
+
* In the center of the screen make sure the profile selected is the **{user} coolstore**
* At the top right of the screen, ensure the __Server Status__ is **Running**
** Click the **Start** button if it isn't and wait till the Server Status is **Running**

. Run MTA analysis:
+
* In the right side of the screen click on the **Run Analysis** button.
+
[NOTE]
====
In this scenario the lead developer is migrating a legacy JavaEE monlithic application into a cloud native Quarkus application to run on OpenShift. +
They want to understand the level of effort needed to migrate a representative application for a "wave" of JavaEE apps to be migrated.
====
+
image::mta-dls/exercise-2/Initial-analysis.jpg[Run Initial Static Code Analysis,link=self,window=blank,title="Run Initial Static Code Analysis"]
+
. Review the analysis results:
+
The **Analysis Results** appear as a listing grouped by __Issues__
+
The analysis identifies:
+
* Required changes to migrate to new frameworks and standards
* Deprecated APIs that need updating
* Configuration patterns that should be modernized
* Container-readiness issues
* OpenShift-specific optimizations
+
. Examine a specific recommendation:
+
Review the result: **@MessageDriven - EJBs are not supported in Quarkus** +
+
image::mta-dls/exercise-2/EJB-issue.jpg[EJB Issue - Not Supported,link=self,window=blank,title="EJB Issue - Not Supported"]
+
* __If this is NOT the first result in the listing do the following:__
** Type __OrderServiceMDB__ in the Search box at the top of the Analysis Results listing
* Click on the **>** arrow on the left of the result and view the details
* A short statement provides more context on the issue and basic guidance
* A link to the affected file is included
* A later exercise uses AI capabilities as Developer Lightspeed conducts an interactive conversation with an LLM to generate code fixes, prompting it with information from rules. This is a RAG-style approach that creates enriched prompts

[NOTE]
====
For the Lead Developer this is a major issue, newer Java frameworks like Quarkus don't support legacy JavaEE EJB (Enterprise Java Bean) approaches.  Especially when working with messaging solutions legacy EJB approaches need to be removed and modern  __MicroProfile__ approaches need to injected into the code.  This also directly reflects the risk assessment reviewed in the first module, where legacy monolithic applications using JavaEE frameworks were a major concern.
====

=== Verify

Confirm the analysis provides actionable insights:

- The MTA extension identified migration issues with the codebase
- Findings are grouped by default by Issues
- Links to code files are included in analysis results
- Remediation suggestions are provided
