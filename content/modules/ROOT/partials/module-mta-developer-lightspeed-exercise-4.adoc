Enterprise JavaBeans (EJBs) were a popular way to build enterprise applications in the past. However, they are not supported in Quarkus. In this exercise we will transform a `ShippingService` from a Stateless Session Bean (EJB) into a modern RESTful API. 

=== The Paradigm Shift: RMI to HTTP
Legacy EJBs were designed for RMI (Remote Method Invocation). This required brittle "Remote Interfaces" and tight coupling between the client and the server.

REST (Representational State Transfer) uses standard HTTP. It is the universal language of the cloud, allowing any service (Python, Go, JS) to communicate with the Java backend without special client libraries. This is a key benefit of cloud-native architecture.

==== Benefits of Going Cloud-Native
* **Decoupling:** Services are found via URLs, not complex JNDI lookups.
* **Observability:** Modern tools like Service Meshes (Istio) can monitor HTTP traffic easily; RMI is largely "invisible" to cloud infrastructure.
* **Simplicity:** Refactoring moves from the complex EJB lifecycle to a simple `@ApplicationScoped` CDI bean.


==== Step-by-Step with Developer Lightspeed
*1*. The first step is to review the Analysis from our previous exercise. In the Analysis panel search box type the class name `ShippingService`. Before we start, ensure that the `Agent Mode` toggle is set to `On`. 

image::mta-dls/exercise-4/shippingservice-analysis-results.jpg[ShippingService,link=self,window=blank,title="ShippingService"]

Lets review the issues reported.
- `@Stateless` and `@Remote` annotations are not supported in Quarkus.
  Enterprise Java Beans (EJBs) are not supported in Quarkus. CDI must be used. Please replace the @Stateless annotation with a CDI scope annotation like @ApplicationScoped.
- `javax.ejb` imports are not supported in Quarkus.
  Replace the javax.ejb import statement with jakarta.ejb
- The package `javax` has been replaced by `jakarta`.
  Replace the javax.jms import statement with jakarta.jms


*2*. Now that we understand the issues of JavaEE to Quarkus migration in context of messaging, lets ask the LLM to generate curated fixes for this migration scenario.

Developer Lightspeed can request fix for a single issue, but it can also request a fix for the entire file. Click on the drop down menu as shown in the image below, and select Files option. This will group all the issues together.

image::mta-dls/exercise-4/shippingservice-analysis-results.jpg[ShippingService,link=self,window=blank,title="ShippingService"]

Next click on the `AI` dropdown as shown in the image below and click `Get Solution`. This will trigger a new tab that will now send the entire context to the LLM. This approach helps us to engineer the context in a way which is more specific to the requirements. A chat-bot would usually be a free flow of questions and messages. Here the context is prepared by Developer Lightspeed based on the static code analysis, in order to get the best results without wasting time or tokens.

Agents will look to fix potential issues once a fix was made.
